# 模拟用户消息库 v2 —— 高阶用户长会话(2026-07-19 草案待审)

**定位**:v1(`user-messages.md`)是"功能点冒烟"——每个 session 5 条
短消息、单意图、无长程状态,测的是"功能在不在"。v2 模拟的是**重度
使用 Claude Code / Codex 的资深工程师**的真实输入,测的是"产品在真实
压力下是否成立"。**未经用户审定前不投产。**

## 用户画像

Staff 级后端工程师,每天 6+ 小时泡在 coding agent 里。特征:

- **消息多意图复合**:一条消息里同时有指令、约束、否定条件和验收标准。
- **默认不信任**:agent 的任何结论都可能被要求出示证据链(命令原始
  输出、文件:行号、git 证据);转述不算数。
- **远程引用**:"20 turn 前我说的那个约束"、"昨天那版日志"——代词和
  省略指代横跨整个会话甚至跨天。
- **随时纠偏**:steer/queue/interrupt 是日常操作,不是异常路径;
  改主意不道歉、不解释。
- **成本与撤销意识**:关心 token 消耗、要求可回滚、要求不可逆操作
  二次确认。
- **口吻真实**:终端上完整句,手机上错字连篇的短句;中英混杂;
  烦了就一个字("停"、"继续"、"结果?")。

## 标注约定

- `(steer)` / `(queue)`:上一条运行中发出,期望立即转向 / 排队到 turn 边界。
- `(interrupt)`:Esc 打断活动后发出。
- `(贴:xxx)`:粘贴附件/长文本,内容按描述现场生成。
- `(隔 N 小时)` / `(次日·resume)`:发送时机,测跨空闲期与 resume。
- `(手机)`:短句、允许错字,模拟移动端输入。
- **加粗的验收注记**:该消息想测出什么,不是用户消息的一部分。

## 与 v1 的差异 / 为什么这些能测出 v1 测不出的问题

| 维度 | v1 | v2 |
|------|----|----|
| 会话长度 | 5 turn | 7–35 turn,含隔夜 |
| 消息意图 | 单意图 | 一条消息 3–5 个动作+否定约束 |
| 指代距离 | 上一条 | 跨 20+ turn、跨 compact、跨天 |
| 对 agent 的态度 | 全信 | 默认要证据,转述不算数 |
| 纠偏 | 偶发 | steer/queue/interrupt 为常态,含方法级纠偏 |
| 状态机压力 | 单一功能 | plan→acceptEdits→goal→babysit→schedule 串联 |
| 失败模式覆盖 | 功能缺失 | 约束遗忘、事实漂移、越权改动、盲从外部意见、粉饰失败 |

**预期能暴露的问题类别**(QA 执行时按此归档 findings):
1. **约束遗忘**:负向指令("别动 api 包"、"OOM 按下不表")在
   compaction/长对话后失效。
2. **事实漂移**:声明与 git/diff/events 真实状态不一致。
3. **语义混淆**:steer 被当 queue、"commit 不 push"被推、
   "留 A 删 C"清理错对象。
4. **证据缺失**:被要求原始输出时给转述或复述记忆。
5. **越权与盲从**:reviewer 意见覆盖用户硬约束;lead 违反
   "只协调"角色约束。
6. **收尾不彻底**:schedule 未真正取消、worktree 未清、
   clear 后旧约束仍生效。

**未覆盖待补**(v3 候选):dictate 真音频、MCP 生态接入、
不受信仓库审计(UJ-20)、崩溃自愈中的用户视角(UJ-21)、
多用户/多客户端同时驾驶同一会话。
