# AgentRunner — 开发流程（PROCESS）

**这是什么**：本项目的产品定义与开发管理方式。核心思想：产品由三层
文档定义（journey → spec → design），需求以**增量**方式进入——一个
需求的三层 delta 完全明确之后才动代码，落地以 journey 级端到端测试
验收。此方式是 v1/v2 两代计划中被验证有效纪律的常态化
（双闸门、一步一提交、决策台账、对抗 review）。

---

## 一、三层产品定义

```
第一层  JOURNEYS.md   端到端 user journey —— 产品要做什么
   ↓ 拆分
第二层  SPEC.md       功能点登记簿 —— 有哪些功能、什么状态、验收锚在哪
   ↓ 衍生
第三层  DESIGN.md     架构 —— 功能点如何成立、为什么这样设计
```

- **第一层回答"为什么存在"**。journey 是端到端的使用故事，定义产品
  边界；一个功能不被任何 journey 覆盖，先怀疑它该不该做。
  第一层有一个**伴生物**：**不变量**（DESIGN §15 已定决策表与各章
  粗体条款）——journey 覆盖"功能"，不变量覆盖"性质"（崩溃不丢输入、
  journal-first、凭据红线这类横切保证，没有任何单条 journey 的情节
  能压到它们）。两者共同构成需求层。
- **第二层回答"有什么"**。功能点从 journey 拆出、按功能域组织、可
  继续向下细分；每条登记：状态、journey 来源、验收锚（哪条测试证明
  它 work）。SPEC 是"正面清单"，GAPS 是它的镜像"负面清单"（审计件）。
- **第三层回答"怎么成立"**。DESIGN 是架构 source of truth，承载语义
  契约与不变量。功能点的实现方式由它裁决。

**支撑件**（不属于三层，服务三层）：

| 文档 | 职责 |
|---|---|
| `QA.md` | journey 级**真实 API** 验收场景菜单（脚本在 `qa/`），第一层测试的可执行形态 |
| `GAPS.md` | 审计件：journey × 设计/实现 的缺口登记（负面清单） |
| `LOG.md` | 增量与决策台账，只追加 |
| `archive/` | 已完成计划与过时文档的封存（只读） |

**冲突裁决顺序**：产品意图以 JOURNEYS 为准 → 实现状态以 SPEC 为准 →
语义契约以 DESIGN 为准（含 §17 实现状态注记的诚实偏差）→ 归档件永远
让位于活文档。发现两份活文档冲突 = 缺陷，当场修（并查第三份），
不带病共存。

## 二、增量开发流程（一个需求怎么进来）

**单位是"增量"**：一个新需求 / 一个 GAPS 缺口 / 一次行为修订。
流程五步，**前三步是文档，后两步才是代码**：

1. **入口**：需求表述为 journey 的增补或修订（新场景 → JOURNEYS 加条
   目或步骤；既有缺口 → 引用 GAPS 条目）。说不清 journey 的需求先退回
   想清楚。
2. **工作纸**：在 `docs/increments/INC-<序号>-<slug>.md` 起草，模板见
   下。工作纸是**临时载体**——落地后并回三层、原件移
   `archive/increments/`，绝不让工作纸与三层长期并存。
3. **裁决**：三层 delta 在工作纸里完全明确（含验收场景与实施步骤），
   开发者确认后才进入实施。**触及 DESIGN 不变量的，先走第四节的
   不变量变更流程。**
4. **实施**：按第三节执行协议逐步落地。
5. **收口**：delta 并回 JOURNEYS/SPEC/DESIGN；QA/GAPS 同步；LOG 记
   一条增量条目（决策、偏差、记档）；工作纸归档。规模达到里程碑级的
   增量做三视角对抗 review（正确性/并发、安全、契约——契约基准 =
   DESIGN + QA），P0/P1 修完才算关闭；小增量可在工作纸里声明裁掉
   review 及理由。

**工作纸模板**：

```markdown
# INC-<n> <标题>
## 动机与 journey 锚          # 为什么做;动/增哪条 UJ 的哪几步
## Spec delta                 # SPEC.md 增改哪些功能点(逐条)
## Design delta               # DESIGN.md 动哪些章节;是否触不变量
## 验收                       # 新增/修订的 QA 场景 + scripted 孪生清单
## 实施步骤                   # 一步=一个可合并提交,标注每步完成标志
## review 裁决                # 做/裁三视角 review,理由
```

## 三、执行协议（实施期纪律，承自 v1 PLAN §0.5 与 v2 PLAN §0）

1. **session 开始**：`git fetch origin main` + fast-forward，永远在
   最新代码上工作。
2. **一步 = 一个可合并提交**：代码 + 测试 + 文档行齐活，
   `./scripts/check.sh` 全绿才 commit；消息格式
   `INC-<n>.<步>: <摘要>`；立即 push `origin/main`（只用 main）。
   不留未推送提交、不留未提交改动过夜。
3. **回归红线**：全部单元测试 + v1 acceptance 26 场景 + 既有 QA 场景
   在每一步保持绿；行为变化一律 opt-in 落地，不破坏既有形态。
4. **双闸门**（每个功能点两条腿，缺一不可）：
   - **闸门 A：scripted 孪生**——确定性、离线（scripted/routing
     provider + crash 注入），进 `check.sh` 常跑；
   - **闸门 B：真实 API QA 场景**——QA.md 菜单逐字照做（断言只钉
     runtime 红线，不钉模型措辞；一次重跑规则适用），结果归档
     `qa/runs/<日期>-<QA号>/`。B 闸失败 → 归因（runtime bug / prompt /
     环境）；runtime bug 必须修复后重跑。
5. **设计冲突**：实现中发现 DESIGN 语义走不通 → **停下**，在 LOG 写清
   冲突，修订 DESIGN 与实现**同一个 commit** 落地——禁止代码悄悄绕。
6. **台账**：影响后续工作的决策/偏差/接受不修，一律记 LOG.md。

## 四、不变量变更流程（硬性）

DESIGN 的不变量 = §15 已定决策表 + 各章粗体条款。动它们必须：

1. **停下**当前实施；
2. 在工作纸里**单独成文**：旧不变量原文 / 为什么必须动 / 新表述 /
   波及面（哪些代码、测试、文档跟着变）；
3. **单独 review**（至少契约视角），通过后 DESIGN 修订与实现同
   commit 落地；
4. LOG 记档。

禁止的路径只有一条，但要写明白：**代码先绕过不变量、文档事后再补**。

## 五、测试体系速览（谁证明什么）

| 层 | 位置 | 证明什么 | 何时跑 |
|---|---|---|---|
| 单元/集成 | `internal/**/*_test.go` | 机制正确 | 每步（check.sh） |
| scripted 孪生 | 同上（scripted/routing provider） | 功能点确定性端到端 | 每步（check.sh） |
| crash 注入 | 同上 | in-doubt/恢复不变量 | 每步（check.sh） |
| acceptance | `e2e/`（26 场景） | v1 各阶段能力回归 | 每步红线 |
| QA 场景 | `qa/run-qa*.sh`（QA.md 菜单） | journey 在**真实 API** 下 work | 增量闸门 B / 收口扫验 |

新增量的验收锚按此分层落位：功能点 → 孪生 + QA 场景；性质 → crash
注入/单测。SPEC 的"验收锚"列必须指到真实存在的测试。

## 六、归档纪律

见 `archive/README.md`。要点：计划关闭/文档被取代即移入 archive 加
头部注记，正文不动、引用不修、永不删除。
